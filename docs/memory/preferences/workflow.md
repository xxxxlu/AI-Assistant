# 工作流程偏好

## 目标

- 快速交付，但保持可追溯、可回滚、可审查。

## 默认执行流程

1. 明确目标与验收标准。
2. 拆分任务并识别风险。
3. 小步实现（优先复用现有组件/hooks/utils）。
4. 本地验证核心路径（至少覆盖验收标准）。
5. 发起 PR 并完成审查。
6. 合并后验证线上/测试环境行为。
7. 必要时补充文档与决策记录。

## Git 流程（唯一口径）

### 分支策略

- 主干：`main`（受保护，禁止直推）
- 功能分支：`feature/*`
- 修复分支：`bugfix/*`
- 紧急修复：`hotfix/*`
- 文档分支：`docs/*`

### 合并规则

1. 所有改动通过 PR 合并。
2. 至少一次有效 Code Review。
3. 合并前必须完成基础自测。
4. 高风险改动必须写明回滚方案。

## 优先级与门禁

- `P0`：线上故障/核心链路。
  最低要求：验收标准 + PR + 回滚方案。
- `P1`：重要功能或高影响改动。
  最低要求：方案说明 + PR + 测试记录。
- `P2`：一般优化。
  按常规迭代节奏处理。

## 文档更新规则

满足任一条件时需要更新文档：

- 路由、规则、偏好发生变化。
- 技术选型或架构决策发生变化。
- 新增可复用流程或约束。

## 沟通偏好

- 结论先行，再给关键依据。
- 风险和前提必须显式说明。
- 若存在不确定项，明确标注并给出下一步。

## 冲突处理

- 若与项目现有流程冲突：项目流程优先。
- 若与系统/开发者更高规则冲突：上层规则优先。
