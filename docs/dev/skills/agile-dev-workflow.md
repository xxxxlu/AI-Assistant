# 技能：敏捷开发与快速响应流程  
(Skill: Agile Development & Fast-Track Workflow)

---

## 1. 概述（Overview）

本技能用于应对 **紧急需求、口头需求或需要快速上线的场景**。  
在不走完整流程的前提下，通过最小化约束与明确底线，确保：

- 开发可以足够快
- 关键质量点不被跳过
- 改动具备可追溯与可回滚能力

> 原则：**快，但不乱；简，但不失控**

---

## 2. 适用场景（Applicable Scenarios）

- **口头需求**：尚未形成完整文档的临时需求
- **紧急修复**：线上问题或高优先级 Bug
- **快速原型**：验证方案或短期试用功能
- **小范围迭代**：影响面可控的局部调整

---

## 3. 变更分级（必须判断）

在进入快线流程前，需对改动进行分级：

- **P0（最高优先级）**
  - 线上故障 / 资金 / 核心链路
  - 允许先处理后补全流程
  - **必须具备回滚方案与发布后验证**

- **P1（高优先级，可控）**
  - 影响有限但需尽快上线
  - 必须 PR + CR 后合并

- **P2（普通改动）**
  - 一般需求 / 体验优化
  - 不建议走快线，按常规流程执行

---

## 4. 核心流程（Fast-Track）

### 阶段一：需求快速确认

- 明确 **核心诉求** 与 **目标**
- 至少定义一句话的 **验收标准**
  - 例：用户点击按钮后，页面提示成功
- 判断是否存在：
  - 接口依赖
  - 字段不稳定

---

### 阶段二：技术方案与开发

#### 接口与数据策略

- **接口已确认**：按契约实现
- **接口未确认或不稳定**：
  - 采用 **Static-first**
  - 基于 UI Model + Mock 数据推进
  - 严禁假设未确认的后端字段

> Mock 与数据处理仍需遵守项目既有分层规范（api / types / views）。

#### 开发约定

- **复用优先**：优先使用已有组件 / hooks / utils
- **可回滚意识**：确保改动具备回退路径
- **自测必做**：本地验证核心功能

---

### 阶段三：代码审查与测试

- **必须通过 PR**
- **Code Review 不可跳过**
  - P0 场景允许事后补 CR，但必须补
- 测试要求：
  - 覆盖验收标准
  - 影响面较大时做最小回归验证

---

### 阶段四：发布与验证

- **若支持灰度**：优先灰度发布
- **若不支持灰度**：
  - 明确回滚方式 / 开关策略
- 发布后：
  - 进行线上验证
  - 关注关键日志与异常

---

## 5. 快线底线（必须满足）

即使在最紧急情况下，也 **不得跳过以下三点**：

1. 明确验收标准
2. 通过 PR（禁止直推主干）
3. 至少一种可回滚手段

---

## 6. 行为准则（AI / 开发者通用）

### 允许

- 基于明确指令创建文件与结构
- 使用 Mock / Static-first 推进开发
- 优化代码结构与可读性
- 补充必要注释与说明

### 禁止

- 编造或猜测业务规则
- 假设后端接口或字段含义
- 跳过验收标准或 CR
- 直接修改主干分支代码

---

## 7. 说明

- 本流程为 **快线补充方案**，不替代标准开发流程
- 是否启用快线，以 **风险与影响面判断** 为准
- 文档以项目实际执行为准，允许根据团队情况微调
